02 · 문제 제기

왜 AI 코딩 에이전트는 백만 줄이 되었나

OpenHands, SWE-agent, Devin 계열의 시스템은 강력하지만, 복잡성의 중심이 실제 코딩보다 오케스트레이션과 제어 계층으로 이동했다는 비판을 받는다.

“LLM 이전 시대엔 필수였던 복잡성의 상당 부분이, 지금은 불필요한 짐일 수 있다.”

기존 방식의 누적

  • 상태 관리가 별도 계층으로 증식한다.
  • 오류 처리가 예외 케이스별 분기 코드로 늘어난다.
  • 플러그인 시스템이 확장성을 위해 구조를 무겁게 만든다.
  • 설정과 어댑터가 여러 모델과 런타임을 감싼다.

강연의 비판 포인트

  • 핵심 가치는 LLM의 실제 작업 수행인데, 주변 인프라 코드가 대부분을 차지한다.
  • 기여 장벽이 커져 코드베이스 이해와 수정이 어려워진다.
  • 유지비용이 올라가고 실험 속도는 떨어진다.
  • 패러다임 미스매치가 발생한다. LLM 시대인데 설계는 여전히 과거 기준이다.
03 · 핵심 개념

NanoClaw는 무엇인가

Gavriel Cohen과 Lazer Cohen이 만든 AI 코딩 에이전트로, 파일 편집, 셸 실행, 코드 수정 같은 핵심 기능을 유지하면서도 3,000줄 수준으로 압축한 사례다.

핵심 주장
기능을 버려서가 아니라, 아키텍처를 AI 네이티브로 다시 설계했기 때문에 작아졌다.

🛠️

핵심 도구는 남긴다

파일 시스템 접근, 셸 실행, LLM 통신처럼 실제 행동을 위해 필요한 최소 도구만 구현한다.

🪶

글루 코드는 줄인다

중간 레이어, 이벤트 버스, 과도한 추상화, 다층 설정 관리 같은 간접 비용을 과감히 제거한다.

♻️

LLM에게 판단을 맡긴다

계획, 오류 복구, 다음 단계 선택을 코드 대신 모델의 추론으로 처리하게 설계한다.

NanoClaw AI 코딩 에이전트 Qwibit 도구 호출 에이전트 루프 경량 아키텍처
04 · 핵심 개념

AI 네이티브 아키텍처의 전환

전통적 소프트웨어는 로직을 코드에 명시하지만, AI 네이티브 소프트웨어는 로직의 상당 부분을 LLM에게 위임한다. 코드의 역할은 ‘제어’에서 ‘도구 제공’으로 이동한다.

프레임
“LLM이 할 수 있는 것은 코드로 만들지 않는다.”

1. 목표 정의

개발자는 원하는 결과와 제약을 자연어로 제공한다.

2. LLM 추론

모델이 계획과 우선순위, 오류 복구 전략을 세운다.

3. 도구 호출

파일 편집, 셸 실행, 브라우징 등 행동을 직접 수행한다.

4. 루프 완료

관찰과 실행을 반복해 작업이 끝날 때까지 스스로 이어간다.

05 · 핵심 개념

1회전 프롬프트는 왜 강력한가

복잡한 플래너, 단계별 상태 머신, 복구 시나리오를 코드로 분리하는 대신, 구조화된 하나의 프롬프트가 작업 전체의 맥락을 쥐도록 만든다.

핵심 이득
복잡성은 코드에서 빠지고, LLM의 자연스러운 추론 흐름 속으로 흡수된다.

기존 오케스트레이션

단계별 플래너, 중간 결과 저장, 예외 흐름 복구, 도구 라우팅, 별도 정책 계층이 각각 코드를 요구한다.

플래너가 다음 행동을 계산
상태 머신이 단계 전환을 관리
실패 핸들러가 재시도 경로를 분기

Single-Turn Prompt 방식

명확한 목표, 허용된 도구, 제약 조건, 완료 기준을 한 번에 제공하고, LLM이 내부적으로 계획과 재계획을 실행한다.

프롬프트가 전략과 경계조건을 제공
LLM이 스스로 작업 순서를 구성
도구 호출 결과를 다시 문맥에 넣어 적응
06 · 주요 내용 전개

전통적 에이전트 vs AI 네이티브

같은 “코딩을 대신하는 에이전트”라도 내부에서 무엇이 복잡성의 중심을 차지하는지가 완전히 다르다.

핵심 대비
과거는 코드가 주도했고, 이제는 모델이 주도한다.

항목 전통적 에이전트 NanoClaw식 AI 네이티브
핵심 중심 오케스트레이션 코드와 상태 제어 LLM 추론 + 최소 도구 레이어
오류 처리 예외 규칙과 분기 로직으로 관리 LLM이 관찰 후 복구 전략을 선택
확장 구조 플러그인, 어댑터, 이벤트 버스 중심 핵심 도구만 두고 필요할 때만 추가
복잡성 위치 코드베이스 전체에 분산 프롬프트 설계와 모델 문맥에 집중
개발 속도 구조가 무거워 실험 비용이 큼 코드가 작아 반복과 수정이 빠름
리스크 유지보수와 진입 장벽 LLM 의존도와 예측 가능성 관리 필요
07 · 주요 내용 전개

설계 원칙: 무엇을 남기고 무엇을 버릴 것인가

NanoClaw의 메시지는 단순한 미니멀리즘이 아니다. 코드와 프롬프트의 역할 분담을 처음부터 다시 설계하자는 제안이다.

재구성된 방법론
기능 나열 → LLM 위임 판단 → 코드 최소화 → 도구 레이어 구현 → 반복 검증

1

핵심 기능 식별

파일 읽기/쓰기, 셸 실행, 모델 호출처럼 실제 작업 수행에 필요한 최소 단위를 분리한다.

2

위임 가능성 판단

이 판단이 LLM의 자연스러운 강점인지, 아니면 강한 결정론이 필요한지 구분한다.

3

코드 최소화

LLM이 처리 가능한 영역에 제어 코드를 추가하지 않는다. 보조 레이어의 폭증을 차단한다.

4

루프 검증

실제 작업 흐름에서 필요한 것만 남기고 반복적으로 제거한다. 간결성은 결과가 아니라 훈련된 선택이다.

09 · 데이터 시각화

복잡성과 위임의 구조를 차트로 보기

정량 수치와 개념 비율을 결합해, 왜 이 강연이 “소프트웨어 2.0의 실용적 구현”으로 읽히는지 시각화했다.

차트 읽는 법
막대는 코드 규모, 도넛은 전통적 복잡성의 분포, 레이더는 AI 네이티브가 강하게 강조하는 축을 나타낸다.

차트 해석 포인트

LOC 격차: 기술적 우월성보다 설계 철학의 차이를 상징한다.
복잡성 분포: 전통적 에이전트는 글루 코드와 오케스트레이션이 부피를 키운다.
레이더 축: AI 네이티브는 위임도, 속도, 단순성은 높이지만 통제성과 예측가능성은 별도 설계가 필요하다.
10 · 주요 내용 전개

무엇이 제거되었나

강연에서 강조되는 제거 대상은 기능 자체가 아니라, 기능을 둘러싸고 점점 두꺼워졌던 보일러플레이트와 추상화 계층이다.

원칙
LLM이 유연하게 흡수 가능한 영역이라면, 그 위에 별도 소프트웨어 구조를 쌓지 않는다.

🧱

다계층 추상화

도구, 실행기, 정책, 라우터를 여러 번 감싸는 래퍼 구조를 축소한다.

🔌

플러그인/익스텐션

일반화를 위해 만든 대규모 확장 프레임워크가 오히려 실험 속도를 늦추는 지점을 문제 삼는다.

🚌

이벤트 버스와 내부 통신

컴포넌트 간 복잡한 메시징 레이어 대신, 더 직접적인 도구 호출과 결과 재주입을 선호한다.

⚙️

방대한 설정 관리

모든 경우를 지원하려는 설정 체계는 코드와 문서 모두를 무겁게 만든다.

🔁

명시적 복구 로직

실패마다 분기 코드를 쓰기보다, LLM이 관찰 후 다음 전략을 택하게 하는 편이 더 자연스럽다는 주장이다.

🧪

과잉 인프라

모니터링, 관찰성, 다중 백엔드 지원도 초기에는 진짜 필요한 수준만 남기는 편이 낫다고 본다.

11 · 주요 내용 전개

최소 아키텍처 스택

NanoClaw식 구조를 추상화하면, 중앙에는 LLM 추론 루프가 있고 그 아래에 필요한 도구들만 얇게 배치된다.

구조 요약
목표 → 프롬프트 → LLM 루프 → 도구 호출 → 피드백 → 완료

사용자 목표 / 작업 요청

자연어로 원하는 기능과 제약 조건을 입력한다.

LLM 추론 코어

계획, 도구 선택, 결과 해석, 오류 복구를 문맥 안에서 연속적으로 수행한다.

도구 레이어

파일 시스템, 셸, 브라우징, 코드 편집 같은 실제 실행 권한을 제공한다.

실행 루프 / 종료 조건

관찰과 행동을 반복하고, 목표 달성 여부를 체크하며 완료된다.
