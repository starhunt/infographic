🎯 SLC 접근 방식: MVP 시대를 넘어서

🚀 Simple, Lovable, Complete

"This PRD defines an SLC simple lovable complete V25.0.1 not an MVP"

⚡

Simple

복잡하지 않고 사용하기 쉬운 직관적인 인터페이스

❤️

Lovable

사용자가 사랑하고 자주 사용하고 싶어하는 매력적인 경험

✅

Complete

세련되고 완성된 느낌의 폴리시된 최종 제품

"Honestly that MVP era it's kind of done. Shipping something that just barely works just doesn't impress anyone anymore. The bar is so much higher now."

개발 기준의 진화

📅 연도 기반 버전 관리

예시: 25.0.0 (2025년 앱스토어 출시 목표)

연도 기반 버전 관리를 통해 개발자 자신의 혼란을 줄이고, AI가 버전 관련 질문으로 방해하지 않도록 합니다.

💬 AI와의 진정한 대화

질문 1: What's the actual problem I'm solving?

해결하고자 하는 실제 문제를 명확히 정의

질문 2: Who are my users really?

구체적인 사용자 페르소나 설정

질문 3: Which features actually matter?

핵심 기능과 부차적 기능 구분

답변을 PRD에 기록

AI와의 대화 결과를 문서에 체계적으로 정리

# Smart NFC Tools - PRD Example ## Problem Statement "iOS NFC tools kind of suck right now. They're clunky, they're dated, and the stuff you actually want is buried like five taps deep." ## Target Users - iPhone automation enthusiasts - Content creators embedding links - Power users building shortcuts - Educators and hobbyists

🎨 사고의 캡처 (Capturing Thinking)

"The whole point isn't to build a feature list. It's to capture the way I'm actually thinking about the app."

PRD는 단순히 기능 목록이 아니라, 애플리케이션에 대한 개발자의 사고와 의도를 담아내는 도구입니다.

💡 핵심 인사이트 & 액션 아이템

1

AI 시대의 PRD는 문서가 아닌 도구다

기존의 형식적 문서에서 AI와 실시간으로 소통하는 살아있는 인터페이스로 패러다임 전환

2

구체성이 AI 성능을 결정한다

모호한 요구사항보다 명확하고 상세한 정의가 AI의 정확한 지원을 가능하게 함

3

SLC는 새로운 기본 기준이다

AI 도구의 발달로 MVP 수준은 더 이상 충분하지 않으며, 완성도 높은 제품이 필수

🚀 즉시 실행 가능한 액션 아이템

  • 현재 프로젝트에 PRD.md 파일 생성하고 SLC 선언문 추가하기
  • AI와의 대화를 통해 구체적인 사용자 페르소나 정의하기
  • Out of Scope 목록을 명시적으로 작성하여 개발 범위 명확화하기
  • Cursor 또는 Claude에 PRD를 규칙으로 추가하여 AI 에이전트 온보딩하기
  • 주간 단위로 PRD 업데이트 및 AI 동기화 루틴 구축하기

🎯 "올바른 질문하기"를 배우는 것

이 PRD 기반 AI 협업 방법론이 초보 개발자에게 제공하는 가장 큰 가치는 "올바른 질문하기"를 배우는 것입니다.

❓

핵심 질문들

"What's the actual problem?", "Who are my users really?"

🧠

제품 사고력

기술이 아닌 사용자 중심의 사고 방식

📋

체계적 접근

막연한 아이디어를 구체적인 명세로 변환

아이디어

"NFC 앱을 만들고 싶다"

구체화

"iPhone 사용자가 3탭 이내로 NFC 태그 작성"

실행 계획

구체적인 UX 플로우와 기술 구현

"예를 들어, '막연한 아이디어'를 'iPhone 사용자가 3탭 이내로 NFC 태그를 작성할 수 있는 직관적인 도구'로 구체화하는 과정을 통해 제품 사고(product thinking)를 배울 수 있습니다."

⚠️ 주의사항

  • AI가 생성한 코드를 이해하지 못한 채 사용하지 말 것
  • 기본적인 디버깅 능력은 별도로 학습 필요
  • 작은 프로젝트부터 시작하여 점진적으로 복잡도 증가
  • PRD는 기본 개발 능력을 대체가 아닌 보완하는 도구

과도한 AI 의존성

기본적인 문제 해결 능력을 잃고 AI에 과도하게 의존하는 위험

📋

PRD 품질의 중요성

잘못된 PRD는 전체 프로젝트를 잘못된 방향으로 유도

🔄

동기화 문제

PRD와 코드베이스 간의 불일치로 인한 혼란

🔴 주요 위험 요소들

  • 이해 없는 코드 사용: AI 생성 코드를 이해하지 못한 채 사용 시 디버깅 불가
  • 보안 취약점: 보안이 중요한 애플리케이션에서 AI 코드에 대한 철저한 검토 필수
  • 기술 부채 누적: 단기적 편의를 위해 장기적 유지보수성 희생
  • 팀 프로젝트 복잡성: 여러 개발자가 동시에 PRD 수정 시 충돌 가능성

위험 요소별 발생 빈도와 영향도

🛡️ 위험 완화 전략

  • 주 1회 AI 없이 코딩하는 날 설정하여 기본기 유지
  • PRD 품질 체크리스트 운영 및 주간 리뷰
  • AI 생성 코드에 대한 이해도 검증 프로세스
  • 보안 취약점 스캔 도구 통합
  • 버전 관리 시스템을 통한 PRD 변경 이력 추적
"PRD가 코드베이스와 동기화되지 않으면 '거짓말하는 문서'가 될 수 있습니다. 특히 팀 프로젝트에서는 별도의 프로세스가 필요합니다."
